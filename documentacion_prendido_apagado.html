API: Disponibilidad de Productos (versión con branchId único y arrays skus[])

1. Especificación del Endpoint

PropiedadValor
MétodoPOST
Endpoint/api/catalog/itemAvailability
Content-Typeapplication/json
AutenticaciónAuthorization: Bearer

2. Formato del Request Body

{
  "branchId": 4390,
  "itemsOn": [
    { "skus": ["2001", "2002"], "platformId": 2 }
  ],
  "itemsOff": [
    { "skus": ["127", "64"], "platformId": 1 }
  ]
}
ClaveTipoReglas
branchIdintegerSucursal destino (un solo valor por request).
itemsOn[] / itemsOff[]arrayCada elemento define: platformId y un array skus[].
skusarrayLista (≥ 1) de SKUs a afectar. Solo dígitos.

Nota: Si necesitas afectar varias sucursales, envía varias peticiones independientes, una por branchId.

3. Formato del Response Body (HTTP 200 OK siempre que el JSON sea válido)

CampoDescripción
status"SUCCESS", "PARTIAL_FAILURE" o "FAILURE".
results[]Lotes exitosos (branchId, platformId, skus[], action, status).
errors[]Lotes fallidos (mismo formato, añade message).
timestampFecha-hora en formato ISO-8601 (UTC).
 
{
  "status": "PARTIAL_FAILURE",
  "results": [
    { "branchId": 4390, "platformId": 2, "skus": ["2001","2002"], "action": "ON",  "status": "SUCCESS" }
  ],
  "errors": [
    { "branchId": 4390, "platformId": 1, "skus": ["127","64"], "action": "OFF", "status": "FAILURE", "message": "Invalid credentials for platformId 1" }
  ],
  "timestamp": "2025-07-01T14:05:00Z"
}

5. Errores de Transporte (HTTP ≠ 200)

CódigoUso Típico
400JSON mal formado, campos faltantes, skus vacío, etc.
401 / 403Token inválido o sin permisos.
429Rate-limit excedido.
500Error interno inesperado.

Ejemplo (400 Bad Request):

HTTP/1.1 400 Bad Request
{
  "error": "itemsOff must be an array",
  "timestamp": "2025-07-01T14:09:05Z"
}

6. Reglas Clave

  • Identificadores únicos: branchId y platformId.
  • Una sola sucursal por request (branchId numérico).
  • itemsOn / itemsOff se procesan por fila (cada fila = plataforma + set de SKUs).
  • Si el JSON es válido → HTTP 200 OK; la lógica de negocio se indica en status.
  • El cliente inspecciona errors[] solo si status ≠ "SUCCESS".
  • Códigos 4xx/5xx reservados a problemas de transporte / validación global.
